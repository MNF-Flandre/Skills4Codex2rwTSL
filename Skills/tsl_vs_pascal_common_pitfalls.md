# TSL 与 Pascal/常见语言迁移易错点（第二遍整理）

## 1. 语言相似度结论（简版）

- 与 Pascal 的总体相似度：约 **70% - 75%**（本次取 **72%**）
- 高相似部分：`begin/end`、`if/then/else`、`case/of`、`for/while/repeat`、类与 `virtual/override` 风格
- 中差异部分：参数传递约定、异常处理、多重继承、数据结构（Array/Matrix/TableArray）
- 低相似部分：金融时序函数、TS-SQL、并发与外部调用

## 2. 高优先级易错点（先看）

1. 字符串拼接用错
- 现象：使用 `&` 导致编译连锁报错（`end not found` 等）
- 正确：优先用 `$`（部分环境也支持 `+`）

2. 赋值与比较混淆
- 现象：把 `:=`、`=`、`==` 混用
- 正确：赋值用 `:=`；比较优先用 `=`（`==` 是否可用看具体环境）

3. `if/else` 分支块写得不明确
- 现象：单行写法叠加嵌套，解析器误配 `else`
- 正确：复杂分支一律显式 `begin ... end`

4. 子类重写忘记 `override`
- 现象：变成“隐藏”而非“覆盖”，多态调用结果不对
- 正确：基类方法 `virtual`，子类重写显式 `override`

5. `for in` 中修改被遍历容器
- 现象：遍历过程中改结构导致行为异常
- 正确：需要修改时，改用索引 `for` 循环

6. `try/except` 异常对象取法错误
- 现象：按 Java/Python 方式写异常形参
- 正确：在 `except` 块通过 `ExceptObject` 读取异常信息

7. 日期/时间表示混用
- 现象：整数日期与日期字面量混搭，比较或转换出错
- 正确：项目内统一一种日期表示与转换路径

8. 指标函数在短序列上直接调用
- 现象：`MA/EMA/RSI` 等输入长度不足，返回 `nil` 或异常
- 正确：先校验历史长度，再计算并做 `nil` 保护

## 3. 详细易错清单（Pascal/C/Python/Java 迁移）

1. 字符串连接：`&` 在部分 TSL 环境不可用，统一用 `$`。
2. 赋值符号：TSL 赋值是 `:=`，不是 `=` 或 `==`。
3. 比较符号：条件判断优先使用 `=`、`<>`。
4. 分支块：嵌套 `if/else` 不要省略 `begin/end`。
5. `case` 误用：不要按 C `switch` 思路写可执行表达式标签。
6. 循环风格：没有 C 经典三段式 `for(i=...;...;...)`。
7. 倒序遍历：使用 `downto`，不要手搓负步长递增逻辑。
8. `for in` 语义：遍历对象/容器时不要改其结构。
9. 参数副作用：注意 `var`/编译指令导致的按引用传递副作用。
10. 异常模型：`try/except/finally` 与传统 Pascal 不同，需专门掌握。
11. 多重继承：同名成员可能歧义，必要时用 `Class(Base).Method()` 指定。
12. 覆盖与隐藏：不写 `override` 可能导致动态分派失效。
13. 默认可见性：类成员默认行为与 Java/C++ 习惯不同，建议显式分段声明。
14. `nil` 处理：对象/数组元素可能为 `nil`，调用前先判空。
15. 类型转换：字符串与数字互转尽量显式，不依赖隐式转换。
16. 浮点精度：金融计算前统一精度策略（例如配合精度设置函数）。
17. Array/Matrix 混淆：构造、索引和函数族不同，不可想当然互换。
18. TS-SQL 误解：`Select ... from ... end` 不是标准 SQL，细节要按 TSL 语法。
19. `RefsOf` 层级：嵌套查询上溯层数写错会引用错数据。
20. 外部调用：`external`/`MakeInstance` 涉及 ABI、位数、参数布局，需严格匹配。
21. 并发误区：`#multirun` 不自动保证共享变量安全。
22. 数据依赖：`Spec(...)` 依赖行情/数据源环境，本地测试要做兜底。

## 4. 对照速查（写代码前看 30 秒）

- 想写赋值：用 `:=`
- 想拼字符串：优先 `$`
- 想写复杂 `if/else`：强制 `begin/end`
- 想做继承重写：`virtual + override`
- 想算指标：先检查序列长度与 `nil`
- 想做子查询引用：先画清 `RefsOf` 的层级

## 5. 简单语法细节高频坑（新增）

1. `if` 后漏写 `then`
- 错误示例：`if a>0 b:=1;`
- 正确示例：`if a>0 then b:=1;`

2. `while` 后漏写 `do`
- 错误示例：`while i<n i:=i+1;`
- 正确示例：`while i<n do i:=i+1;`

3. `for` 赋值符写成 `=`
- 错误示例：`for i=1 to n do ...`
- 正确示例：`for i:=1 to n do ...`

4. 倒序循环不用 `downto`
- 错误示例：`for i:=n to 1 do ...`
- 正确示例：`for i:=n downto 1 do ...`

5. 嵌套分支省略 `begin/end` 导致 `else` 归属错误
- 建议：只要出现嵌套分支，就显式写 `begin ... end`。

6. `case` 语句写成 C 风格
- 错误示例：`case x { ... }`
- 正确示例：`case x of ... end;`

7. `repeat ... until` 写成 `while` 风格
- 错误示例：`repeat ... end;`
- 正确示例：`repeat ... until cond;`

8. 语句结尾漏分号 `;`
- 现象：在 `End` 附近报连锁语法错误。
- 建议：逐行检查赋值、调用、`return`、块结束后的分号。

9. `begin/end` 与 `Begin/End` 混杂且缩进混乱
- 说明：关键字通常不区分大小写，但混写会影响可读性和排错效率。
- 建议：统一使用一种风格（例如 `begin/end`）。

10. 布尔与空值字面量拼写错误
- 建议：统一使用 `true/false/nil`，不要写成其他语言的 `True/False/None/null`。

11. 函数头和函数体分隔符不完整
- 错误示例：`Function f(a) Begin ... End;`
- 正确示例：`Function f(a); Begin ... End;`

12. 复杂表达式里把 `:=` 当作比较
- 建议：条件里只放比较表达式，赋值单独成句，避免“边赋值边判断”的写法。

13. 字符串引号风格不一致
- 说明：文档示例同时出现单引号和双引号。
- 建议：项目里固定一种主风格；含引号内容时再按需切换。

14. 忘记在函数中显式 `return`
- 现象：函数执行后返回空值或非预期默认值。
- 建议：每个函数至少有一条明确的返回路径。

15. 语句块层级不清导致 `end not found`
- 建议：写完一个块立即补 `end;`，再回填中间逻辑，减少括块不平衡。

## 6. 建议的最小编码规范

1. 全项目统一字符串拼接符（推荐 `$`）。
2. 任何嵌套分支必须带 `begin/end`。
3. 所有重写方法必须写 `override`。
4. 金融计算入口统一做：长度校验 + `nil` 防护 + 精度设定。
5. 涉及 `external`、`#multirun` 的代码必须单独评审。
